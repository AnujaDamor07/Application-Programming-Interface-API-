# 🚦 Rate Limiting

**Rate limiting** caps how many requests a client can fire at an API within a given time window. It's the main line of defense against abuse, traffic spikes, and automated attacks like brute-forcing or denial-of-service attempts.

By keeping request frequency in check, rate limiting helps an API stay **available, stable, and fair to all its users**.

---

## ❓ Why It Matters

### 1️⃣ Stops DoS/DDoS Attacks
Without a real rate limit in place, an attacker can hammer an API with thousands of requests per second, exhausting server resources and taking the service down.

### 2️⃣ Protects What's Behind the API
Most APIs touch databases, file systems, or other external services. A flood of requests can overwhelm those just as easily as the API itself.

### 3️⃣ Keeps Usage Fair
Without limits, a single client can soak up all available capacity, starving everyone else out.

### 4️⃣ Keeps the System Stable
Controlling the inbound flow of requests keeps the API responsive for the people actually using it normally.

### 5️⃣ Slows Down Brute Force Attempts
Login endpoints and other sensitive operations become much harder to brute-force once repeated attempts get throttled.

---

## ⚙️ How It's Usually Implemented

Rate limiting generally comes down to three pieces:

### 1️⃣ A Request Threshold

The max number of requests allowed.

**Examples:**
- 100 requests per minute
- 10 requests per second

### 2️⃣ A Time Window

The period over which those requests get counted — per second, minute, hour, or day.

### 3️⃣ An Identifier to Apply the Limit Against

Limits can be tied to:
- 🌐 IP address
- 🔑 API key
- 👤 User account
- 🎫 Session token
- 🪪 JWT token

---

## 🌐 A Real Pattern Worth Watching For

Some APIs expose a parameter that lets the client control how many times a backend action gets repeated — which can quietly bypass whatever rate limiting exists elsewhere.

**Example endpoint:**
```http
POST /workshop/api/merchant/contact_mechanic HTTP/1.1
Host: <target-host>
```

**Example body:**
```json
{
  "mechanic_code": "<mechanic_code>",
  "problem_details": "test",
  "vin": "<vehicle_vin>",
  "mechanic_api": "http://<target-host>/workshop/api/mechanic/receive_report",
  "repeat_request_if_failed": true,
  "number_of_repeats": 100
}
```

### ⚠️ The Risk

The `number_of_repeats` field here lets the client tell the backend to fire off that same internal action multiple times in a row — potentially sidestepping rate limiting that only checks the *incoming* client request, not what happens server-side afterward.

If the backend takes this parameter at face value, an attacker can:
- 💥 Generate a flood of internal requests from a single external call
- ⚙️ Trigger resource exhaustion
- 🌊 Flood internal services that were never meant to be hit this hard

---

## 🧪 How to Test for It

### 1️⃣ Flood It and See What Happens

Fire a burst of requests at the target endpoint in a short window and watch how the API responds.

**Tools commonly used for this:**
- 🐝 Burp Suite Intruder
- ⚡ Turbo Intruder
- 🎯 ffuf
- 🐍 Custom Python scripts

**What you're checking:** does the API actually block excessive requests, or does it just keep accepting them indefinitely?

### 2️⃣ Watch the Response Carefully

Signs that rate limiting is working:
```
HTTP 429 Too Many Requests
```

Or headers like:
```
X-RateLimit-Limit
X-RateLimit-Remaining
Retry-After
```

### 3️⃣ Test Whether It's Tied to the Right Identifier

Check whether the limit is keyed to:
- 🌐 IP address
- 🪪 JWT token
- 👤 User account

A weak implementation might let an attacker just spin up multiple tokens or sessions and sidestep the limit entirely.

### 4️⃣ Test for Loop-Style Parameter Abuse

Look specifically at parameters like:
```
number_of_repeats
repeat_request_if_failed
```

and check whether they can trigger uncontrolled backend request loops or extra internal calls beyond what the client actually sent.

---

## 🚨 Impact When It's Missing or Weak

Without solid rate limiting, attackers can:
- ⚠️ Abuse the API at will
- 🔓 Run brute-force attacks
- 💥 Trigger denial-of-service conditions
- ⚙️ Exhaust backend services
- 📡 Spin up excessive internal API traffic

### 📉 Which Can Lead To
- ⚠️ Service degradation
- 🔌 API downtime
- 💰 Higher infrastructure costs

---

## 🛡️ How to Fix It

### 1️⃣ Enforce Rate Limiting Server-Side

**Example limits:**
- 10 requests per second
- 100 requests per minute, per user

Never rely on the client to self-limit — enforce it where the client can't tamper with it.

### 2️⃣ Apply Both Global and Per-User Limits

Limits should cover:
- 🌐 IP address
- 👤 User account
- 🔑 API key

That way attackers can't just spin up new accounts or tokens to dodge the cap.

### 3️⃣ Never Trust Client-Controlled Loop Parameters

Don't let client-supplied fields like:
```
number_of_repeats
repeat_request_if_failed
```

drive how many times the backend repeats an internal action — that decision belongs entirely server-side.

### 4️⃣ Throttle Requests Properly

Once a client exceeds its limit, respond with:
```
HTTP/1.1 429 Too Many Requests
```

and a header like:
```
Retry-After: 60
```

so the client knows exactly how long to back off.

### 5️⃣ Put an API Gateway or WAF in Front of It

Options like:
- 🌐 API gateways
- 🔁 Reverse proxies
- 🛡️ Web Application Firewalls

These can enforce rate limiting *before* a request ever reaches the backend, adding a layer of protection independent of the application's own logic.

---

## 📌 Wrapping Up

Rate limiting is one of those controls that's easy to overlook until it's missing — but it's essential for keeping an API resilient against abuse, brute-force attempts, and denial-of-service conditions.

Get it right, and you get:
- ⚖️ Fair resource usage across all clients
- ⚙️ A more stable, predictable system
- 🔐 Real protection against automated attacks
- 🌐 An API that stays reliably available
