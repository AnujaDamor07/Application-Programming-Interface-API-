# 🧪 VAmPI — Lab Setup & Testing Notes

**Repository:** https://github.com/erev0s/VAmPI

VAmPI stands for **Vulnerable API**, built on Flask. It's purpose-made to walk through the **OWASP API Security Top 10** inside a clean, easy-to-read REST setup.

---

## ⚙️ Installation Options

> Either approach below works fine on its own — no need to do both.

### 🚀 Option 1 — Run the Prebuilt Image
```bash
docker run -d -e vulnerable=1 -e tokentimetolive=3000 -p 5000:5000 erev0s/vampi:latest
```

Then check it's live at:
- `http://<your-ip>:5000/`
- `http://<your-ip>:5000/ui/`

### 🐳 Option 2 — Clone & Run via Docker Compose
```bash
cd /opt
git clone https://github.com/erev0s/VAmPI.git
cd /opt/VAmPI
docker-compose up -d
```

### ✅ Confirm It's Running
```bash
docker ps
netstat -nltup
```

## 🏗️ Architecture at a Glance

VAmPI is:
- A REST API
- Built on Flask
- Authenticated via JWT
- A single container — much simpler than crAPI
- Backed by SQLite or Postgres, depending on configuration

## 🔎 Initial Recon Strategy

### Step 1 — Work Through Swagger

Visit:
```text
http://<your-ip>:5000/ui/
```

Go through:
- Every listed endpoint
- Required parameters
- What authentication each one needs
- Response schemas

This alone maps out most of the attack surface.

### Step 2 — Find the Authentication Flow

Usually it's:
```text
POST /users/v1/login
POST /users/v1/register
```

Grab the JWT it returns and pick apart:
- The header
- The payload
- The signature

## 5️⃣ Vulnerabilities Built Into VAmPI

### 1. Broken Object Level Authorization (BOLA)

```text
GET /users/v1/users/1
GET /users/v1/users/2
```

Swap the user ID and see whether data leaks across accounts.

### 2. JWT Weaknesses

**Check for:**
- A hardcoded signing secret somewhere in the source
- A weak signing algorithm
- Whether role claims can be tampered with

**Try:**
- Changing `role` to `admin`
- Changing `user_id`
- Re-signing the token if you've recovered the secret

### 3. Mass Assignment

When registering or updating a user:
```json
{
  "username": "rahul",
  "password": "test",
  "admin": true
}
```

Try slipping in extra fields like:
- `is_admin`
- `role`
- `balance`

### 4. Excessive Data Exposure

Watch for responses that look like:
```json
{
  "id": 1,
  "username": "admin",
  "password": "hashedvalue",
  "secret_key": "..."
}
```

Look out for:
- Internal-only fields
- Debug information
- Sensitive data that has no reason to be in the response

### 5. Rate Limiting Gaps

Try a login brute-force:
```bash
ffuf -X POST -d '{"username":"admin","password":"FUZZ"}' \
-u http://<your-ip>:5000/users/v1/login \
-w rockyou.txt
```

Check whether:
- The account ever locks out
- Any rate limiting kicks in
- Responses differ between valid and invalid attempts

### 6. Injection Testing

Try:
- SQL injection in parameters
- Command injection in search fields
- JSON-based injection

## 🧭 A Sensible Pentest Flow

A clean order to work through this lab in:

1. Recon via Swagger
2. Register an account
3. Capture the JWT
4. Then test for:
   - BOLA
   - JWT manipulation
   - Mass assignment
   - Injection
   - Role tampering
   - ID enumeration
   - Rate limiting
   - Error handling behavior

## 📊 VAmPI vs vAPI vs crAPI

| Feature          | vAPI     | VAmPI   | crAPI     |
|--------------------|----------|---------|-----------|
| Difficulty         | Low      | Medium  | High      |
| JWT focus          | Limited  | Strong  | Advanced  |
| Microservices       | No       | No      | Yes       |
| Business logic      | Minimal  | Moderate| Advanced  |
| Swagger docs        | No       | Yes     | Partial   |

VAmPI is a great fit for:
- Practicing JWT exploitation
- Getting solid at BOLA
- Learning how API authorization actually breaks

## 🚀 Going Further

For a deeper dive:
- Read through the source (`app.py`, `config.py`)
- Spot any hardcoded secrets
- Trace how authorization decorators are applied
- Understand exactly how the JWT gets validated

Once you understand the logic, you can attack based on actual flaws in that logic rather than just blind testing.
