# 🔍 Endpoint Discovery & API Fuzzing Methodology

Finding endpoints is arguably the most important phase of any API pentest. Miss an endpoint, and you miss whatever vulnerability was sitting behind it.

This guide walks through practical examples using **OWASP Juice Shop** as the target.

---

## 1️⃣ Manual Endpoint Discovery

### 1.1 Walk Through the App Yourself

- Click through every feature you can find
- Register and log in
- Update your profile
- Add items to a cart
- Run through checkout
- Try reaching the admin panel, if there is one

**Use a proxy while you do this:**
- Burp Suite
- OWASP ZAP

**Capture as you go:**
- Hidden API calls
- JSON responses
- Auth tokens
- Endpoints tied to specific roles

### 1.2 Dig Through the JavaScript

Worth checking:
- `/assets/*.js`
- `/main.js`
- `/runtime.js`

Search inside them for:
```
/api/
/rest/
fetch(
axios(
http.get(
```

JS bundles frequently reveal:
- Endpoints that aren't documented anywhere
- Admin-only routes
- Old/deprecated API versions
- API paths hardcoded directly into the frontend

## 2️⃣ Fuzzing the API

**Useful wordlist repo:**
https://github.com/chrislockard/api_wordlist

```bash
git clone https://github.com/chrislockard/api_wordlist.git
```

These wordlists are organized around:
- Objects
- Actions
- Paths

### 2.1 Fuzzing with ffuf

```bash
ffuf -u http://target.com/FUZZ -w api_wordlist/common.txt
```

Try fuzzing across multiple prefixes:
- `/api/FUZZ`
- `/rest/FUZZ`
- `/v1/FUZZ`
- `/admin/FUZZ`

## 3️⃣ Testing Every HTTP Method

Run every method against each endpoint you discover.

| Method   | What It's For      | What to Test For       |
|-----------|---------------------|--------------------------|
| GET       | Reading data        | IDOR                     |
| POST      | Creating             | Mass assignment          |
| PUT       | Updating             | Privilege escalation     |
| DELETE    | Removing             | Unauthorized deletion    |
| PATCH     | Partial updates      | Logic abuse               |
| HEAD      | Header-only check    | Info leakage              |
| OPTIONS   | Allowed methods      | Misconfiguration          |
| TRACE     | Echo/diagnostic       | XST (rare, still worth checking) |
| CONNECT   | Proxy tunneling       | Rare, but worth a check  |

**Example:**
```bash
curl -X PUT http://target.com/api/users/1
```

**Check for:**
- `200` where you'd expect a `403`
- Method-override headers
- Methods that should be blocked but aren't

## 4️⃣ Using Postman Alongside This

### 4.1 Testing Token-Based Auth

Example token:
```
eyJhbGciOiJSUzI1NiJ9...
```

**Decode it and check:**
- The signing algorithm (e.g. RS256)
- Key claims like `sub`, `role`, `exp`

**Then test:**
- Changing `role` to `admin`
- Stripping the signature entirely
- Modifying the expiration
- Replaying an expired token

## 6️⃣ More Advanced Discovery Techniques

### 6.1 Probe for Other Versions

Try:
```
/v1/
/v2/
/v3/
/beta/
/internal/
```

Older versions tend to:
- Skip auth checks that newer versions enforce
- Still have debug features switched on

### 6.2 Parameter Discovery

```bash
ffuf -u http://target.com/api/users?FUZZ=test -w params.txt
```

Look for:
- Parameters that aren't documented
- Debug flags
- Things like `admin=true`
- Things like `role=admin`

### 6.3 GraphQL Discovery

Check for:
```
/graphql
/graphiql
```

And test whether introspection is left enabled.
