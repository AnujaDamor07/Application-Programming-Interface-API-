# 🔐 Broken User Authentication

## 📖 What It Is

Broken User Authentication shows up whenever an app fails to properly lock down its authentication or account-recovery flows — opening the door for attackers to take over accounts that aren't theirs.

A common variant of this: the **password reset flow doesn't actually check that the OTP being verified belongs to the email address in the request**.

That gap lets an attacker reset *someone else's* password by submitting a valid OTP alongside a *different* email address — effectively hijacking that OTP for an account it was never issued for.

⚠️ End result: full **unauthorized account takeover**.

---

## ⚙️ Where This Shows Up

### 🔑 The Password Reset Flow
```
http://<target-host>/forgot-password
```

### 🌐 The Vulnerable API Call
```
POST /identity/api/auth/v2/check-otp
```

---

## 🧪 How to Test for It

### Step 1 — Kick Off a Password Reset

Go to:
```
http://<target-host>/forgot-password
```
Request an OTP for an account you control.

### Step 2 — Intercept the OTP Verification Request

Catch the request in a proxy like Burp Suite.

**Example request:**
```http
POST /identity/api/auth/v2/check-otp HTTP/1.1
Host: <target-host>
Content-Type: application/json
```

**Example payload:**
```json
{
  "email": "your-test-account@example.com",
  "otp": "1234",
  "password": "NewPassword123!"
}
```

### Step 3 — Swap the Target Account

Keep the valid OTP, but change the `email` field to point at a different account:

```json
{
  "email": "victim@example.com",
  "otp": "1234",
  "password": "NewPassword123!"
}
```

If this succeeds, it proves the **OTP was never actually bound to the account it was issued for** — meaning anyone holding a valid OTP can apply it to a completely different user.

### What Happens If It's Vulnerable

⚠️ The password for the victim's account gets reset, and the attacker now controls it.

---

## ✅ What Should Happen Instead

The server needs to verify all of the following before accepting an OTP:
- 🔗 The OTP actually matches the email it was issued for
- 🔒 The OTP is tied to the exact session that originally requested the reset
- 🚫 The OTP can't be reused for any other account

It should respond with something like:
```
403 Forbidden
```
or
```
Invalid OTP
```
whenever those checks fail.

---

## 🚨 Why This Matters

If exploited, this lets an attacker:
- 🔑 Reset the password of basically any user
- 👤 Take over their account outright
- 📂 Access whatever sensitive data sits behind that account
- ⚙️ Act as that user inside the application

**Depending on the app, that can cascade into:**
- 🚗 Unauthorized tracking of someone's vehicle
- 📜 Access to their order/service history
- 🧾 Exposure of personal information
- 🛠️ Abuse of whatever functionality the account has access to

## 🧩 Root Cause

The password reset mechanism never ties the OTP to the original request and the specific account it was generated for.

Instead of validating the `email` field against the actual OTP-issuance record, the API just **trusts whatever email is submitted alongside the OTP**. That trust gap is exactly what lets an attacker swap in a different email and reuse a valid OTP against someone else's account.

---

## 🛠️ How to Fix It

### 1. Tie the OTP to the Account It Was Issued For

Store the OTP alongside the user it belongs to, not as a standalone value.

**Example structure:**
```
email | otp | expiry | used
```

**On verification, the server should check:**
- The OTP matches the one stored for *that specific* email
- It hasn't expired
- It hasn't already been used
- It belongs to the original reset session — not just any session

This closes off the ability to reuse one account's OTP against another.

### 2. Use Secure Reset Tokens Instead of Raw OTP Checks

**A more robust flow looks like:**
1. User requests a password reset
2. Server generates a unique, single-use reset token
3. That token is bound directly to the requesting account
4. The token expires quickly

### 3. Enforce a Short OTP Expiry

```
OTP expiry = 5 minutes
```

A tight expiry window shrinks the attack surface dramatically.

### 4. Add Rate Limiting

```
Max 5 OTP attempts per account
```

This blocks brute-forcing OTP values outright.

### 5. Invalidate the OTP Immediately After Use

Once a reset succeeds, the OTP should be marked used (or deleted) so it can never be replayed.

### 6. Log and Monitor for Abuse

Watch for patterns like:
- 📌 Repeated OTP attempts on the same account
- 🔄 Reset attempts hopping across many different accounts
- 🌐 Requests coming from unusual or unexpected IPs
- ⚠️ A string of failed OTP verifications in a row

Good logging here is often what catches an attack in progress before it succeeds.
