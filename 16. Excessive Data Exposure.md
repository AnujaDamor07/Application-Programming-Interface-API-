# 📤 Excessive Data Exposure

## What It Is

**Excessive Data Exposure** happens when an API hands back more data than the client app actually needs. Instead of filtering sensitive fields out on the **server side**, the API dumps the full object and just trusts the client to display only the relevant parts.

The problem is, nothing stops a curious (or malicious) user from inspecting the raw response directly — bypassing whatever filtering the frontend was supposed to do — and seeing everything the server actually sent.

During API pentesting, you spot this by carefully going through every API response and checking whether it includes fields that shouldn't be there: internal IDs, auth tokens, debug info, or personal user details that have no business being in that particular response.

If exploited, this can lead to information disclosure, privacy violations, and can act as a stepping stone toward bigger attacks like privilege escalation or account takeover.

---

## 🔍 Why This Is Worth Testing For

APIs sit right in the middle between frontend apps and backend databases, which makes them a high-value target — get the response filtering wrong, and sensitive data leaks straight through.

Testing for this matters because:

- 🔐 It can **leak confidential user data** — personal details, account IDs, email addresses, auth tokens.
- 👤 Exposed data gives attackers material to **impersonate users, escalate privileges, or run social engineering attacks**.
- 📊 It can enable **mass user enumeration or large-scale data harvesting**, raising the risk of a much bigger breach.
- ⚙️ Debug or internal-only endpoints can reveal **system configuration, internal API structure, or leftover development artifacts** — all useful intel for further attacks.

---

## 🌐 Where This Tends to Show Up

A few categories of endpoint are worth paying close attention to during testing:

- **Social/community feed endpoints** — e.g. something like `/community/api/v2/community/posts/recent`, which can leak more user metadata than the UI actually displays.
- **User listing endpoints** — e.g. `/users/v1`, which sometimes return full user records instead of a trimmed-down public profile.
- **Debug-only endpoints** — e.g. `/users/v1/_debug`, often left enabled accidentally and capable of dumping internal state.
- **Personal media/content endpoints** — e.g. `/identity/api/v2/user/videos`, which may expose content or metadata tied to other users.

> ⚠️ Treat any of these as a red flag if they return more fields than the client application actually consumes — especially internal identifiers, tokens, or anything resembling debug output.

---

## 🛠️ How to Test for It

1. Walk through the application's normal functionality while intercepting traffic with Burp Suite or a similar proxy.
2. For every API response, compare what's **rendered in the UI** against what's **actually returned in the JSON**.
3. Flag anything extra: internal IDs, password hashes, tokens, role flags, debug fields, timestamps tied to backend infrastructure, etc.
4. Specifically probe for debug-style paths (`/_debug`, `/internal`, `/test`) that may have been left enabled.
5. Check whether sensitive fields differ based on the requesting user's role — sometimes excessive exposure only shows up for certain account types.

---

## 🛡️ How to Fix It

- **Filter on the server, not the client.** Build explicit response schemas (DTOs) for each endpoint instead of serializing entire database objects.
- **Apply field-level whitelisting.** Only return fields the client genuinely needs — never the full underlying model.
- **Disable or remove debug endpoints** in any environment that isn't strictly internal/development.
- **Review responses by role.** Make sure higher-privilege fields never leak into responses served to lower-privilege users.
- **Audit regularly.** As APIs evolve, new fields get added to models all the time — re-check response payloads periodically to make sure nothing extra has slipped through.
