# 🔓 Broken Object Level Authorization (BOLA)

## What It Is

**Broken Object-Level Authorization (BOLA)** happens when an application fails to check whether the *currently logged-in user* actually owns the object they're requesting — and instead just trusts whatever identifier shows up in the URL or parameters (think `vehicle_id`, `order_id`, `report_id`, etc.).

Because the ID is user-suppliable, an attacker can simply swap it out for someone else's ID and, if no ownership check exists, walk straight into another user's data.

In a typical vulnerable app, you'll find several endpoints that:
- Accept an object ID directly in the path or query string
- Never actually verify that the requesting user is the owner of that object

The end result: an authenticated, otherwise "legitimate" user can pivot sideways into other users' sensitive data just by changing an ID.

---

## Where This Tends to Show Up

This pattern (commonly seen in labs like crAPI) usually appears across a few categories of endpoint:

### 1. Vehicle/Asset Location APIs

```
GET /identity/api/v2/vehicle/{vehicle_id}/location
```

**Example request:**
```http
GET /identity/api/v2/vehicle/<vehicle_id>/location HTTP/1.1
Host: <target-host>
Authorization: Bearer <user_token>
```

#### Steps to Reproduce
1. Log in as **User A**.
2. Open the dashboard and trigger a request that loads vehicle location data.
3. Intercept that request — it'll include a `vehicle_id` (often a UUID).
4. Swap the `vehicle_id` for one that belongs to **User B**.
5. Resend the request.

#### What Happens
The server hands back **User B's vehicle location data**, even though User A has no business seeing it.

#### What Should Happen Instead
The server should respond with:
```
403 Forbidden
```
or an equivalent access-denied response, whenever the requested object doesn't belong to the authenticated user.

---

### 2. Orders APIs

```
GET /workshop/api/shop/orders/{order_id}
```

#### Steps to Reproduce
1. Log in as **User A**.
2. Capture a request for one of your own orders, e.g. `GET /workshop/api/shop/orders/<your_order_id>`.
3. Change the `order_id` to a different value that isn't yours.
4. Resend the request.

#### What Happens
The response includes **order details belonging to a different user entirely**.

---

### 3. Mechanic / Diagnostic Reports

```
GET /workshop/api/mechanic/mechanic_report?report_id={id}
```

#### Steps to Reproduce
1. Log in as a normal, non-privileged user.
2. Capture a request that fetches a mechanic report.
3. Cycle through different `report_id` values.
4. Resend each modified request.

#### What Happens
Reports belonging to **other users** come back in the response, with no ownership check blocking access.

---

## Why This Matters

A successful BOLA exploit can expose:
- Vehicle/location tracking data
- Other users' order histories
- Diagnostic/mechanic reports
- Potentially personally identifiable information (PII)

Depending on the app, this can lead to:
- Privacy violations
- Real-world tracking of another person's vehicle or movements
- Broader data leakage
- Abuse of business logic built on top of these objects

---

## Typical Severity

**High** — this is consistently one of the most impactful and most common API vulnerabilities, which is why it sits at #1 on the OWASP API Security Top 10.

### Example CVSS Scoring

## How to Fix It

### 1. Check Ownership on Every Request

Never assume the ID alone is enough — always confirm the requester actually owns the object.

```python
if order.user_id != current_user.id:
    return 403
```

### 2. Centralize Authorization Logic

Build a shared middleware/decorator layer that every sensitive endpoint runs through, instead of scattering ad-hoc checks across the codebase.

### 3. Avoid Predictable, Guessable IDs

Where possible, move away from sequential or easily-enumerable IDs, and pair that with strict server-side ownership checks regardless.

### 4. Enforce Ownership at the Query Level

Bake the ownership check directly into the database query, not just into application logic that's easy to forget:

```sql
SELECT * FROM orders
WHERE order_id = ?
AND user_id = current_user_id;
```

### 5. Log and Watch for Abuse Patterns

Keep an eye out for:
- A user rapidly cycling through many different object IDs
- Repeated requests for objects that clearly don't belong to that user

These patterns are a strong signal that someone is actively probing for BOLA.
