# ⚠️ Mass Assignment

## What It Is

**Mass Assignment** crops up when an API automatically maps whatever the client sends straight onto internal object properties — with no validation or per-field restrictions in the way.

That opens the door for an attacker to control attributes they were never supposed to touch.

This usually traces back to frameworks that auto-bind request parameters to backend model objects, where developers never bothered to set up an **allowlist** or per-field authorization rules.

---

## 🎯 What an Attacker Can Get At

Once Mass Assignment is possible, slipping extra parameters into a request payload can let an attacker manipulate things like:

- 👤 User roles or permissions
- 🚩 Account status flags
- 🆔 User identifiers
- 📦 Object ownership fields
- ✔️ Verification flags
- ⚙️ Internal configuration values

If the backend processes these without checking, the attacker walks away with:
- 🔓 Privileged access they shouldn't have
- 🛠️ The ability to modify other users' data
- 🚫 A way around security controls entirely

---

## 🧪 How to Hunt for It

### 1️⃣ Understand How the API Actually Maps Data

Look at how request parameters flow into backend objects:
- Go through the API docs
- Study request/response shapes
- Note which fields appear to be expected vs. unusual

### 2️⃣ List Out Likely Sensitive Fields

Try to identify hidden or sensitive attribute names, like:
- `role`
- `is_admin`
- `isVerified`
- `user_id`
- `account_status`
- `ownerID`

These are exactly the kind of fields that shouldn't be user-controllable, even if the API quietly still accepts them.

### 3️⃣ Try Sending Fields That Shouldn't Be There

Add parameters the original request or UI form never included, and see if the API just... accepts them.

- Inject hidden/sensitive fields into the payload
- Compare behavior with and without the extra field

### 4️⃣ Check What Actually Happened

Look at the response and the app's behavior afterward to see whether the injected field was applied for real.

**Signs it worked:**
- ✏️ The value shows up modified in the response
- ⬆️ Your account suddenly has elevated privileges
- 🔄 The change persists across later requests

Any of those confirms the backend is processing fields it really shouldn't be.

## 🧪 Example Scenarios

### 🔺 Example 1 — Privilege Escalation

**Request:**
```http
POST /api/updateProfile HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "username": "test_user",
  "email": "user@example.com",
  "role": "admin"
}
```

**Vulnerable response:**
```json
{
  "status": "success",
  "message": "Profile updated successfully",
  "user": {
    "username": "test_user",
    "email": "user@example.com",
    "role": "admin"
  }
}
```

**What's going on:** the API takes the `role` field at face value, even though it was never meant to be user-editable. That's a direct route to privilege escalation and confirms there's no field-level validation at all.

---

### 🔺 Example 2 — Unauthorized Account Modification / Takeover

**Request:**
```http
PUT /api/updateAccount HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "userID": "12345",
  "username": "attacker",
  "password": "newpassword",
  "email": "attacker@example.com"
}
```

**Vulnerable response:**
```json
{
  "status": "success",
  "message": "Account updated successfully",
  "user": {
    "userID": "12345",
    "username": "attacker",
    "email": "attacker@example.com"
  }
}
```

**What's going on:** the API accepts a client-supplied `userID`, even though that value should be derived from the authenticated session, not the request body. This gives an attacker:
- 🔓 The ability to edit *any* account just by swapping `userID`
- 🔑 The ability to reset other users' passwords
- 📧 The ability to hijack email addresses on other accounts
- ⚠️ All of this because there's no check that the caller actually owns this account

This is Mass Assignment stacked with **Broken Object Level Authorization (BOLA)**.

---

### 🔺 Example 3 — Password Enumeration

**Request:**
```http
POST /api/authenticate HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "username": "test_user",
  "passwords": [
    "password123",
    "letmein",
    "admin@123",
    "password"
  ]
}
```

**Vulnerable response:**
```json
{
  "status": "success",
  "message": "Authenticated successfully with password: letmein"
}
```

**What's going on:** the API accepts an array of candidate passwords in one request, and then — worse — tells the attacker exactly which one worked. That combination gives an attacker:
- 🔐 A fast way to enumerate valid passwords
- 🛠️ A massive speed boost for brute-forcing
- ⚠️ All from a parameter (`passwords`) that should never have been accepted in the first place

This pairs Mass Assignment with **Excessive Data Exposure**.

### 🔺 Example 4 — Modifying Hidden Attributes

**Request:**
```http
PATCH /api/users/12345 HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "username": "test_user",
  "email": "user@example.com",
  "isVerified": true
}
```

**Vulnerable response:**
```json
{
  "status": "success",
  "message": "User updated successfully",
  "user": {
    "userID": "12345",
    "username": "test_user",
    "email": "user@example.com",
    "isVerified": true
  }
}
```

**What's going on:** `isVerified` is meant to be a server-controlled internal flag, but the API happily lets a client flip it directly. That means:
- ⚠️ Verification can be bypassed entirely
- 🚫 Whatever security process verification was meant to enforce gets skipped
- 📧 Email verification specifically can be sidestepped
- 🔓 An unverified, potentially malicious account gets the same trust level as a properly verified one

---

### 🔺 Example 5 — Changing Object Ownership

**Request:**
```http
PUT /api/updateOrder HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "orderID": "67890",
  "userID": "12345",
  "status": "completed"
}
```

**Vulnerable response:**
```json
{
  "status": "success",
  "message": "Order updated successfully",
  "order": {
    "orderID": "67890",
    "userID": "12345",
    "status": "completed"
  }
}
```

**What's going on:** `userID` here should be immutable and tied to whoever actually created the order — but the API lets it be reassigned through the request body. That opens up:
- 🔓 The ability to claim ownership of someone else's order
- 📦 Privilege escalation over business-critical objects
- 💰 Real financial fraud potential, depending on what "ownership" controls
- ⚠️ A clear failure to check object ownership before allowing the update

This combines Mass Assignment with **BOLA** once again.

## 💥 Overall Impact

Left unchecked, Mass Assignment can lead to:
- 🔓 Privilege escalation
- ✏️ Unauthorized data modification
- 👤 Full account takeover
- 🛠️ Business logic bypass
- 🚫 Unauthorized access to other users' resources

## 🛡️ How to Fix It

### 1️⃣ Allowlist Fields Explicitly
Only accept the exact fields each endpoint is supposed to receive.
- Define a whitelist per endpoint
- Reject anything outside that list
- Document clearly which fields a client is allowed to set

### 2️⃣ Enforce Real Authorization Checks
Don't just check that a user is logged in — check that they're allowed to touch *this specific field* on *this specific object*.
- Verify permission per field, not just per endpoint
- Apply role-based access control for sensitive attributes
- Confirm ownership before allowing any modification

### 3️⃣ Separate DTOs From Internal Models
Don't bind request bodies directly onto your database models.
- Build dedicated request/response DTOs
- Map explicitly from DTO to internal model, field by field
- Never let raw client input touch domain objects directly

### 4️⃣ Treat Sensitive Fields as Read-Only
Block modification entirely for fields like:
- `role` / `is_admin`
- `isVerified`
- `accountStatus`
- `userID`

These should be excluded from client-side binding altogether.

### 5️⃣ Validate Everything, Server-Side
- Check field types and value ranges
- Make sure immutable fields genuinely can't be changed
- Run authorization checks on every sensitive operation, not just the obviously risky ones

## 🧪 Real Examples From the crAPI Training Lab

These came up while testing **crAPI**, and show how loose validation and unrestricted parameter binding turn into real Mass Assignment and business-logic bugs.

### 🔺 1. Getting an Item for Free

**The issue:** the API lets you control financial parameters like `quantity` and `credit` directly. Submit a negative quantity, and the price calculation logic breaks in your favor.

**Request:**
```http
POST /workshop/api/shop/orders HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{
  "product_id": 2,
  "quantity": -10,
  "credit": 25500
}
```

**Why it works:**
- The API never validates that `quantity` must be positive
- A negative value flips the order-total math in the attacker's favor
- The result: items obtained without the balance actually decreasing

**Impact:**
- 🎁 Free purchases
- 🛠️ Bypassed pricing logic
- 💰 Direct revenue loss for the business

---

### 🔺 2. Inflating Your Account Balance

**The issue:** an attacker can directly set an order's `status` field to trigger refund logic.

**Request:**
```http
PUT /workshop/api/shop/orders/21 HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{
  "quantity": 1,
  "status": "returned"
}
```

**Why it works:**
- The API lets users update `status` directly, with no restriction
- Setting it to `returned` fires the refund process
- That refund inflates the attacker's balance with no legitimate return having happened

**Impact:**
- 💸 Fraudulent refunds on demand
- 🪙 Balance manipulation with zero real activity behind it
- 💰 Direct financial fraud

---

### 🔺 3. Tampering With Internal Video Processing Parameters

**The issue:** the API exposes internal processing configuration — like `conversion_params` — directly to the client.

**Request:**
```http
PUT /identity/api/v2/user/videos/102 HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{
  "videoName": "Sample_Video_1M.mp4",
  "conversion_params": "-v codec test"
}
```

**Why it works:**
- An internal-only parameter is exposed to the client
- If this value gets passed straight into a system command (e.g. FFmpeg), it opens the door to command injection
- That's effectively letting users send arbitrary instructions into backend infrastructure

**Impact:**
- ⚙️ Direct control over backend media-processing behavior
- 🐛 Potential command injection / arbitrary code execution
- 🔒 An attacker effectively hijacking part of the processing pipeline

## 🔑 Key Takeaways

The crAPI examples above illustrate a recurring set of issues:
- ⚠️ Mass Assignment
- 🚫 Missing input validation
- 🧠 Business logic flaws
- 🔐 Internal parameters exposed where they shouldn't be

These show up consistently across the **OWASP API Security Top 10**, and they're some of the most common — and most damaging — bugs found during real-world API pentests.
