# 🔓 Broken Function Level Authorization (BFLA)

## What It Is

**Broken Function Level Authorization (BFLA)** happens when an API doesn't properly check whether the calling user is actually *allowed* to use a given function or endpoint.

The result: a low-privilege user can reach functionality that should be locked down to higher-privileged roles, like admins. If the authorization check is missing or implemented poorly, an attacker can call privileged backend operations directly — no admin panel needed, just the right (or wrong) HTTP request.

This sits on the **OWASP API Security Top 10** as one of the more critical risk categories, because the impact tends to scale directly with whatever the privileged function actually does.

---

## ⚠️ How It Tends to Happen

### 1️⃣ No Real Role Check
The API never verifies that the authenticated user actually holds the required role or permission before running a sensitive action.

### 2️⃣ Endpoints That Are Too Permissive
Routes that were clearly meant for admins or other privileged users end up reachable by any logged-in account.

### 3️⃣ Authorization Enforced Only on the Frontend
The app hides admin buttons in the UI but never re-checks permissions on the backend — so the restriction disappears the moment someone bypasses the UI and talks to the API directly.

### 4️⃣ Admin Endpoints Left Reachable
Administrative routes exist and respond normally if an attacker just sends a request straight to them, with no extra gating in place.

---

## 🎯 Example Vulnerable Endpoint

```
DELETE /identity/api/v2/admin/videos/{id}
```

```
http://<target-host>/identity/api/v2/admin/videos/<video_id>
```

This looks like an admin-only function for deleting videos from the platform.

---

## 🧪 How to Test for It

Send the request using a **regular user's JWT** (i.e. a token with `role: user`, not an admin token):

```http
DELETE /identity/api/v2/admin/videos/<video_id> HTTP/1.1
Host: <target-host>
Authorization: Bearer <USER_JWT_TOKEN>
Content-Type: application/json
```

**Decode the JWT and confirm the role claim is non-admin:**
```json
{
  "sub": "<user_identifier>",
  "role": "user",
  "iat": 1732182096,
  "exp": 1732786896
}
```

If the request still succeeds despite `role: user`, that confirms the endpoint isn't actually checking permissions server-side — a clear BFLA finding.

---

## 🧾 Why This Is a Problem

A path like:
```
/identity/api/v2/admin/videos/{id}
```

clearly lives under an "admin" namespace, which strongly implies it should require admin-level privileges. If the server doesn't actually verify that before processing the request, any authenticated user — regardless of role — ends up able to invoke functionality meant only for administrators.

---

## 🚨 What an Attacker Can Do With This

- 🗑️ Delete content (videos, in this example) they have no business touching
- ⚙️ Perform other administrative actions
- 🛠️ Modify or remove sensitive data
- 🔌 Disrupt the platform's normal operation
- 🔓 Abuse any other privileged API function discovered the same way

If several admin-namespace endpoints share this flaw, an attacker could end up with something close to full administrative control — all without ever needing actual admin credentials.

---

## 🔥 Severity

**High**

**Why:**
- Direct path to privilege escalation
- Admin-level functionality reachable by anyone
- Strong potential for data manipulation or service disruption
- No special tooling needed — just a normal API request

---

## 🚩 Common Red Flags

Watch for endpoint paths containing:
```
/admin/
/management/
/internal/
/superuser/
```

And keep an eye out for:
- Admin-style functionality that responds normally to a regular user's token
- An absence of any explicit server-side role check in the code or behavior

Either of these strongly suggests privileged operations might be reachable by users who shouldn't have access.

---

## 🛠️ How to Fix It

### 1️⃣ Implement Real Role-Based Access Control (RBAC)

The API has to check the user's role before running anything privileged.

```python
if user.role != "admin":
    return 403  # Forbidden
```

### 2️⃣ Enforce Authorization Server-Side, Always

Never lean on the frontend to "hide" privileged actions — that's not a security control, it's a UI decision. Every authorization check needs to live in the backend, where a client can't bypass it.

### 3️⃣ Lock Down Admin Routes Explicitly

Anything under `/admin/*` (or equivalent) should require admin-level authorization, checked on every single request to that namespace.

### 4️⃣ Centralize Authorization in Middleware

Run every incoming request through shared authorization middleware before it touches business logic. This gives you:
- Consistent enforcement across every endpoint
- A much lower chance of forgetting a check somewhere
- One place to update security logic instead of dozens

### 5️⃣ Log and Monitor Privileged Actions

Anything destructive or sensitive — deleting resources, modifying critical data — should be logged and actively monitored. Good logging here is often what catches abuse early, before it escalates further.

---

## 🔐 What a Secure Response Looks Like

If a regular user hits this endpoint, the correct response is:
```http
HTTP/1.1 403 Forbidden
{
  "error": "Unauthorized access"
}
```

---

## 📌 Summary

When an API doesn't enforce function-level authorization properly, users without the right privileges can still reach administrative functionality just by calling the right endpoint directly.

Left unaddressed, that opens the door to:
- 🔓 Unauthorized administrative actions
- ⚠️ Privilege escalation
- 🛠️ Data manipulation or deletion
- 💥 Broader system compromise, depending on what else is exposed the same way
