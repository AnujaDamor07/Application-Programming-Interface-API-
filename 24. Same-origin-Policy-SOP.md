# 🌐 SOP — Same-Origin Policy

## What It Is

The **Same-Origin Policy (SOP)** is a security rule enforced by the browser itself. It controls whether a script or document loaded from one origin is allowed to interact with resources sitting on a *different* origin.

---

## 🌐 What Counts as an "Origin"?

An origin is the combination of three things:
```
Protocol + Domain + Port
```

Change any one of those, and you've changed origins.

### Examples
- `https://example.com` ≠ `http://example.com` — different protocol
- `https://example.com` ≠ `https://api.example.com` — different subdomain
- `https://example.com:443` ≠ `https://example.com:8080` — different port

---

## ✅ What SOP Allows Within the Same Origin

- Reading data from the same origin
- Modifying the DOM of the same origin
- Accessing cookies and localStorage belonging to the same origin

## 🚫 What SOP Blocks Across Origins

- Reading the actual response body from a different origin
- Touching the DOM of a page on another origin
- Pulling cookies or storage data that belong to another origin

---

## 🔓 Exceptions — Where Cross-Origin Behavior Is Still Allowed

### 1. Loading External Resources

This still works across origins, even though SOP exists:
```html
<script src="https://cdn.example.com/script.js"></script>
<img src="https://target.com/image.png">
<link href="https://cdn.com/style.css">
```

The script runs and the image/stylesheet renders — but the *content* of the response generally isn't something your own JS can read directly.

### 2. Cross-Origin Form Submissions

```html
<form action="https://target.com" method="POST">
```

You can submit a form to a different origin, and the browser will send the request — but your page can't read back whatever response comes from that other origin.

### 3. CORS (Cross-Origin Resource Sharing)

Servers can deliberately loosen SOP's restrictions using response headers like `Access-Control-Allow-Origin`. This is the *intended* escape hatch for legitimate cross-origin API access — but a sloppy CORS configuration is also one of the more common ways SOP protections get accidentally undone.

### 4. Cross-Origin Embedding

Certain elements are still allowed to embed cross-origin content:
- `<iframe>`
- `<img>`
- `<video>`

But even here, SOP still applies where it counts: JavaScript on the parent page can't reach into an iframe's content if that iframe is loaded from a different origin.

---

## 🛡️ Notes for Pentesters

SOP exists specifically to stop one origin from quietly reading another origin's data. That said, several well-known attack classes exist precisely because they find ways around (or exploit gaps next to) SOP:

- Cross-Site Scripting (XSS) — runs attacker JS *inside* the target's own origin, sidestepping SOP entirely
- CORS misconfiguration — a server accidentally trusts origins it shouldn't
- Cross-Site Request Forgery (CSRF) — abuses the fact that requests can still be *sent* cross-origin, even if the response can't be read
- Clickjacking — abuses cross-origin embedding via iframes to trick users into interacting with a page they can't actually see clearly

---

## 📌 Summary

- SOP is a browser-level security boundary built around `Protocol + Domain + Port`.
- Same-origin access is unrestricted; cross-origin *reading* of responses, DOM, cookies, and storage is blocked by default.
- Loading external resources and submitting cross-origin forms are both still allowed — the browser just won't let your script read the response unless CORS explicitly permits it.
- A misconfigured CORS policy, or careless use of cross-origin embedding, can quietly undo SOP's protections and open up real security risk for both web apps and the APIs behind them.
